# Plan — Cleanup & Optimize: Pivot from Live ICS → Curated Supabase

## Context

We just landed an end-to-end live ICS feed (Azure Function proxy → `useIcsEvents` → Header city switcher) but immediately pivoted: events will live in Supabase from now on, fed by a one-shot import script (`scripts/import-ics.mjs`) and curated by hand. The calendar should read from Supabase, not the live feed.

This pass:

1. Adds a `city` column to the events table so the Boston/NYC switcher remains meaningful end-to-end.
2. Tears down the live ICS runtime infrastructure that's about to be dead code.
3. Sweeps orphan files, dead deps, and the pre-existing broken header test.
4. Measures the bundle after cleanup and lists deferred perf wins (no scope-creep into a perf PR).

After this pass, the calendar will be empty until you run the importer + approve events in Supabase. That's the intended state — clean slate, curated content.

---

## Phase A — Schema migration: add `city` to `events`

### A1. New SQL file: `Docs/sql queries/add_city_column.sql`

```sql
-- Idempotent: safe to re-run.
alter table public.events
  add column if not exists city text check (city in ('boston', 'new-york-city')) default 'boston';

create index if not exists events_city_idx on public.events (city);
```

You run this once in the Supabase SQL editor. Default `'boston'` keeps any pre-existing rows valid.

### A2. Type: extend `DatabaseEvent` in [src/types/events.ts](src/types/events.ts)

Add `city: City;` to the interface and import the `City` type from the context. Pass it through in `databaseEventToScheduleX` (calendar doesn't need it directly, but keeping it on the in-memory shape lets future filters/badges read it without a refetch).

## Phase C — Wire Supabase as the source, filtered by city

### C1. [src/hooks/useSupabaseEvents.ts](src/hooks/useSupabaseEvents.ts)

Accept `city: City` as a parameter. Add `.eq("city", city)` to the existing query so each city renders independently. Re-key the `useEffect` on `city` so a switcher click refetches.

### C2. [src/hooks/useEvent.ts](src/hooks/useEvent.ts)

Swap the body back to:

```ts
import { useCity } from "../contexts/CityContext";
import { useSupabaseEvents } from "./useSupabaseEvents";

export function useEvents() {
  const { city } = useCity();
  return useSupabaseEvents(city);
}
```

Net effect: `CityContext` and the Header switcher stay; the data source changes from live ICS → Supabase.

### C3. [src/pages/SubmitEventPage.tsx](src/pages/SubmitEventPage.tsx)

Add a `city` field to the form (radio buttons or `<select>`). Default to whatever `useCity()` currently shows. Include `city: form.city` in the Supabase insert. This keeps user submissions consistent with the new schema.

### C4. [scripts/import-ics.mjs](scripts/import-ics.mjs)

Already created. Update so the row builder sets `city: <feed key>` for each event. One-line change to `veventToRow`.

## Phase G — Bundle measurement + deferred perf

After Phases A–F, run `npm run build` and capture chunk sizes. Expected wins **for free** from this cleanup:

- `ical.js` (~50 KB minified) drops out of the browser bundle entirely (was pulled into `CalendarPage` via `useIcsEvents`).
- `@schedule-x/ical` removal trims a small amount.

Compare against current baseline (`CalendarPage 240K`, `index 316K`, `SubmitEventPage 173K`) and write the new numbers into the verification section below.

**Deferred perf candidates** (do not implement in this pass — flag for a follow-up PR):

- `temporal-polyfill/global` is loaded eagerly via [src/types/events.ts](src/types/events.ts) line 3. It's ~70 KB. Modern Safari/Chrome have `Temporal` behind flags but not enabled. If the polyfill is only used in calendar paths, scoping the import to the calendar route would shave the index chunk.
- `@schedule-x/*` is dynamically imported via `lazy(() => import("./CalendarPage"))` already (good), but the SubmitEventPage chunk is 173K — likely Supabase client + form lib. Worth checking if Supabase can be deferred until form submit.

## Sequencing note

Phase A (SQL migration) **must run in Supabase before** Phase C2/C3 hit production, otherwise queries with `.eq("city", city)` will fail. Order of operations:

1. Run `add_city_column.sql` in Supabase SQL editor.
2. Land Phases A–G in code, deploy.
3. Run the importer to populate.
4. Approve events in Supabase.

If you'd prefer to land code first and run SQL after, gate Phase C1 behind a feature check or temporarily skip the `.eq` call — but the simplest path is "migration first, code second."
